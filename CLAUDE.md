### Deploy = hot-copy (image is stale)
```
cd frontend && npm run build && docker cp build/. cp-api:/app/frontend/build/   # frontend
docker cp app/<f>.py cp-api:/app/app/<f>.py                                       # backend
```
**Gotchas (each bit us this session):**
1. **`kill -HUP 1` graceful reload does NOT re-run FastAPI lifespan.** It reloads workers but lifespan startup (migrations, daemons, init hooks) keeps old in-memory state. A broken lifespan import PASSES an HUP reload and only crashes on **full container restart**. ‚Üí After deleting/renaming backend modules, validate with a full restart, never just HUP.
2. **Hot-copies are ephemeral.** Image is stale vs the running container. `docker compose up -d --force-recreate cp-api` reverts to the old baked image and **wipes all hot-copies** (font, AgentFlow, prune ‚Äî everything this session). DO NOT recreate cp-api until the image is rebuilt.
3. **Durable** = `docker compose build cp-api` (bakes current source). Only then is force-recreate safe.
4. **Chat endpoint is form-encoded** (`-F "message=..."`), not JSON. `app/projects.py:843` reads `form.get("message")`.
5. Routers in `app/main.py` are **try/except guarded** ‚Üí deleting a guarded router file just unregisters it (boot-safe). But **lifespan-body imports are NOT guarded** (e.g. `init_sharepoint` at line ~292) ‚Äî those crash startup.

### Fresh-deploy hardening (AWS cold start) ‚Äî 2026-06-08
Three cold-state-only failures (warm local volumes hid them, fresh AWS volumes hit them). All fixed:
1. **`knowledge/` volume permission crash (the big one).** `knowledge/` is gitignored ‚Üí absent on a clean clone ‚Üí `/app/knowledge` never in the image ‚Üí `chown -R dash:dash /app` skips it ‚Üí Docker creates the fresh `knowledge_data` mountpoint **root-owned** ‚Üí non-root `USER dash` can't write ‚Üí `main.py:1818 makedirs('/app/knowledge/_decks')` ‚Üí `PermissionError` ‚Üí **every worker exit 3 ‚Üí restart loop**. LANDMINE: rebuild does NOT re-chown an EXISTING root-owned volume ‚Äî drop it (`docker volume rm <proj>_knowledge_data`, it's empty) or `chown -R 999:999`.

### Pharma `article_code` ‚Äî cross-table type mismatch + source corruption
The catalog‚Üîstock join key is typed **differently per table** and the stock copy is often **corrupt at source**. Two landmines:

Catalog‚Üîstock then can't link. Fix = re-upload `balance_stock` with `article_code` as **Text** in Excel (or CSV straight from source, never opened in Excel); codes must read `1000000131948`, not `1E+12`. ### Training pipeline + data rules (2026-06-09, latest+49)
- **Aggregate in SQL, never in-head.** `dash/instructions.py` has a HARD RULE: totals/counts/distinct MUST come from SQL `SUM`/`COUNT(DISTINCT)`/`GROUP BY`, never by summing fetched rows (the agent used to undercount many-row queries). Prefer the pre-aggregated `shop_flat` for branch-stock totals.
- **Code/ID columns land as TEXT.** `_is_id_colname` (`app/upload.py`) normalizes separators (`"Article Code"` ‚Üí `article_code`) BEFORE the id/code regex, so codes never land as int64 and cross-table joins don't hit the bigint‚Üîtext mismatch. Never raw-compare `article_code` across tables ‚Äî always `::text`, or use `shop_flat` (already normalized).
- **Derived tables are NOT trained.** `shop_flat` is excluded from the training table list (`_DERIVED_TABLES` in `app/upload.py`); it's built by `build_shop_flat` in post-hooks, not trained. `STOCK_COLS=("site_code","stock_qty","article_code")` in `dash/tools/table_sync.py` ‚Äî `article_code` is REQUIRED so the resolver can never pick `shop_flat` (which keys on `art_key`) as the stock table.
- **Training step timeouts + watchdog.** Relationship/QA verify engines run with `statement_timeout=30s` (`connect_args options`) so an un-indexable `::text` JOIN can't hang a run. `_reap_stale_runs(slug)` (`app/learning.py`) runs on every status poll: any `running`/`finalizing` run with no log progress > `STALE_RUN_MINUTES` (default 12) is auto-failed ‚Üí UI spinner can't spin forever. Run status: `running` ‚Üí `finalizing` (post-hooks) ‚Üí `done`.
- **No ML step.** AutoML removed 2026-05-23; `_task_ml()` is a no-op that PURGES any stale `ml_auto_create` row, frontend filters `ml`/`ml_auto_create`/`ml_models`. No "ML Models" badge, no machine-learning training in this product.
- **`APIGW_AUTO_PROVISION=0`** (default off, `.env` + compose). Loading multi-store stock no longer auto-mints `svc:outlet-*` API service accounts. Turn on only for per-store gateway keys.
- **Fresh-DB baseline** (`db/baseline/schema.sql` recon block): runs under `SET search_path = public, dash;` + `ADD COLUMN IF NOT EXISTS engine_model` on usage tables before the `v_usage_unified` recreate ‚Äî required or a fresh deploy fails to load. Locked `dash_projects` row still seeded by hand on a fresh DB (`ensure_locked_project()` not built).

### Auto-train robustness + replace/sync data-safety guards (2026-06-13, uncommitted)

Triggered by the floating robot showing ‚öÝ "1" ‚Äî auto-train runs aborted `stale: no progress > 12 min` with `logs=[]`, `tables_trained=0` (hung at startup). All backend except `FloatingRobot.svelte`; all env-flagged, defaults safe; verified live on :8011 with a throwaway `smoke_guard` table. See memory `project_citypharma_train_sync_guards`.

**Auto-train (`app/upload.py` `retrain_project._bg`, `app/learning.py`, `dash/cron/auto_train_daemon.py`):**
- **Per-table timeout** `RETRAIN_TABLE_TIMEOUT_S` (default 600s) ‚Äî replaced `for f in as_completed(_futs)` (blocked forever on a hung future ‚Üí run went stale w/ empty logs) with submit-order iteration + `f.result(timeout=‚Ä¶)`; on `TimeoutError` logs `‚äò table X timed out`, counts failed, continues. **`_tpool.shutdown(wait=False, cancel_futures=True)`** so an orphaned hung worker can't re-block at the (old) `with`-pool exit. LANDMINE: don't restore the `with ThreadPoolExecutor()` context-manager form ‚Äî it joins all threads on exit and re-introduces the hang.
- **Startup heartbeat** `_master_log("‚ñ∂ retrain started ‚Äî N table(s)‚Ä¶")` placed AFTER the `_master_log`/`_set_step` closures are defined (placing it right after the master-run INSERT = `NameError`, closures not yet bound). Guarantees `logs` is never `[]` ‚Üí a hang in the pre-table phase (orphan-purge / eval-clear) is now diagnosable.
- **Watchdog surfaces why** `_reap_stale_runs` now also `SET error=` the stale reason (was only `current_step`, invisible to `get_run_status`/robot, keeps any real prior error). `/auto-train/status` `recent_runs` returns `error`+`current_step`. `FloatingRobot.svelte` console shows `‚öÝ last run failed: {error||current_step}`.
- **Circuit-breaker** `AUTO_TRAIN_MAX_RETRY` (default 2) ‚Äî the daemon already auto-retried hung tables via untrained-detection; the real gap was INFINITE retry of a broken table-set (worse now: per-table timeout lets the run finish 'done' but the table stays untrained ‚Üí re-fires every poll). `_untrained_attempts: {slug:(frozenset(tables), n)}`; same set tried > max ‚Üí `action="circuit_open"`, skip trigger, log warn; clears when set changes or `_get_untrained_tables` empties. Exposed in `get_daemon_status().circuit_open`. In-memory (redeploy = 1 fresh attempt).
- **Robot badge** (`FloatingRobot.svelte`) no longer latches forever: `showErr` = failed AND `!errDismissed` AND `!errStale` (>6h `ERR_TTL_H`) AND `!isTrainingNow`; `errDismissed` auto-resets via `$effect` on a new `lastRun.id`; ‚úï dismiss button in `.fr-trainerr` (`.fr-errx`).

**Replace / S3-sync guards** (`app/upload.py` `upload_file` REPLACE branch, NEW `guarded` param resolved from query+form; `app/s3_sync.py` POSTs `guarded=1` ‚Äî manual UI stays flexible, unattended sync is protected):
- **#1 empty file** ‚Äî refuse replace of an existing table with a 0-row file ‚Üí `400` (a parse-layer `400 "File is empty or has no columns"` fires first for header-only too = double safety). Sync's existing `status‚â•400 ‚Üí continue` skips the etag write ‚Üí corrected file re-syncs.
