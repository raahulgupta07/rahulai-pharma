### Deploy = hot-copy (image is stale)
```
cd frontend && npm run build && docker cp build/. cp-api:/app/frontend/build/   # frontend
docker cp app/<f>.py cp-api:/app/app/<f>.py                                       # backend
```
**Gotchas (each bit us this session):**
1. **`kill -HUP 1` graceful reload does NOT re-run FastAPI lifespan.** It reloads workers but lifespan startup (migrations, daemons, init hooks) keeps old in-memory state. A broken lifespan import PASSES an HUP reload and only crashes on **full container restart**. вЖТ After deleting/renaming backend modules, validate with a full restart, never just HUP.
2. **Hot-copies are ephemeral.** Image is stale vs the running container. `docker compose up -d --force-recreate cp-api` reverts to the old baked image and **wipes all hot-copies** (font, AgentFlow, prune вАФ everything this session). DO NOT recreate cp-api until the image is rebuilt.
3. **Durable** = `docker compose build cp-api` (bakes current source). Only then is force-recreate safe.
4. **Chat endpoint is form-encoded** (`-F "message=..."`), not JSON. `app/projects.py:843` reads `form.get("message")`.
5. Routers in `app/main.py` are **try/except guarded** вЖТ deleting a guarded router file just unregisters it (boot-safe). But **lifespan-body imports are NOT guarded** (e.g. `init_sharepoint` at line ~292) вАФ those crash startup.

### Dead RELATED-QUESTIONS chip вАФ payload dropped in wrapper (2026-06-04)
Symptom: clicking a "Related questions" chip on the chat answer card did nothing. Root cause = 3-hop action chain that drops the 2nd arg:
- `AnswerCard.svelte:571` chip вЖТ `onAction('related', q)` (q = question text).
- `ChatMessageList.svelte` wrapper (`onAction={(act, payload) => {...}}`, ~471) mapped followup/diary/copy/pin/save/excel/email/share but **had no `related` branch** вЖТ fell to default `onAction?.(act)` which **dropped `payload`**.
- `+page.svelte:792` `handleAction('related', arg)` needs `arg`=question вЖТ got `undefined` вЖТ empty `q` вЖТ silent `return`.
- Fix: in the wrapper added `if (act === 'related') { onAction?.('related', payload); return; }` AND changed the default bubble `onAction?.(act)` вЖТ `onAction?.(act, payload)` so any arg-aware handler keeps its payload.
- **Rule:** an intermediate `onAction` re-dispatcher MUST forward the payload by default, not just for the cases it special-cases. A bare `onAction?.(act)` silently breaks every arg-carrying action.

### Pharma `article_code` вАФ cross-table type mismatch + source corruption
The catalogвЖФstock join key is typed **differently per table** and the stock copy is often **corrupt at source**. Two landmines:
- **Type mismatch (code, FIXED):** `articles_list.article_code` = **bigint**, `balance_stock.article_code` = **text** (ingest keeps ID cols as string to survive Excel mangling). Raw `b.article_code = a.article_code` = `text = bigint` вЖТ Postgres *"operator does not exist"* вЖТ tool crash вЖТ agent says *"data type mismatch"*. **Always** cast: join on `a.article_code::text`; build `article_code IN (вА¶)` lists as **quoted text literals** (or pass params `str(code)`); normalise result keys back to `int`. Done in `pharma_shop_tool`/`pharma_graph_tool`/`pharma_chemist_tool`. Don't ALTER the text col вЖТ bigint (values may be non-numeric, below).
- **Scientific-notation corruption (DATA, unfixable in code):** Excel writes large IDs as `1E+12`. If the source file was saved in Excel pre-upload, **every** `balance_stock.article_code` can collapse to the single string `"1E+12"` (seen: 102,107 rows вЖТ 1 distinct). CatalogвЖФstock then can't link. The **P3 linkage guard** (`COUNT(*) вА¶ article_code IN (вА¶)` == 0) catches it вЖТ `stock_linkable:false` + `linkage_warning`; agent says *"stock unavailable вАФ linkage broken"* (NOT false "out of stock"). **Correct behavior.** Fix = re-upload `balance_stock` with `article_code` as **Text** in Excel (or CSV straight from source, never opened in Excel); codes must read `1000000131948`, not `1E+12`. Resolver auto-picks new table вЙ§30s.
