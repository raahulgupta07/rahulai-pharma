### Deploy = hot-copy (image is stale)
```
cd frontend && npm run build && docker cp build/. cp-api:/app/frontend/build/   # frontend
docker cp app/<f>.py cp-api:/app/app/<f>.py                                       # backend
```
**Gotchas (each bit us this session):**
1. **`kill -HUP 1` graceful reload does NOT re-run FastAPI lifespan.** It reloads workers but lifespan startup (migrations, daemons, init hooks) keeps old in-memory state. A broken lifespan import PASSES an HUP reload and only crashes on **full container restart**. → After deleting/renaming backend modules, validate with a full restart, never just HUP.
2. **Hot-copies are ephemeral.** Image is stale vs the running container. `docker compose up -d --force-recreate cp-api` reverts to the old baked image and **wipes all hot-copies** (font, AgentFlow, prune — everything this session). DO NOT recreate cp-api until the image is rebuilt.
3. **Durable** = `docker compose build cp-api` (bakes current source). Only then is force-recreate safe.
4. **Chat endpoint is form-encoded** (`-F "message=..."`), not JSON. `app/projects.py:843` reads `form.get("message")`.
5. Routers in `app/main.py` are **try/except guarded** → deleting a guarded router file just unregisters it (boot-safe). But **lifespan-body imports are NOT guarded** (e.g. `init_sharepoint` at line ~292) — those crash startup.

### Fresh-deploy hardening (AWS cold start) — 2026-06-08
Three cold-state-only failures (warm local volumes hid them, fresh AWS volumes hit them). All fixed:
1. **`knowledge/` volume permission crash (the big one).** `knowledge/` is gitignored → absent on a clean clone → `/app/knowledge` never in the image → `chown -R dash:dash /app` skips it → Docker creates the fresh `knowledge_data` mountpoint **root-owned** → non-root `USER dash` can't write → `main.py:1818 makedirs('/app/knowledge/_decks')` → `PermissionError` → **every worker exit 3 → restart loop**. Fix: **Dockerfile pre-creates `/app/knowledge/{_decks,embed_logos,seeds,tables,business,queries,rules}` + `/app/.gunicorn` BEFORE chown** so a fresh volume inherits `dash:dash`; `/decks` makedirs+mount made **fail-soft** in main.py; entrypoint best-effort `mkdir -p`. LANDMINE: rebuild does NOT re-chown an EXISTING root-owned volume — drop it (`docker volume rm <proj>_knowledge_data`, it's empty) or `chown -R 999:999`.
2. **`cp-init` seeder exit 2 on cold DB.** Healthcheck was `pg_isready -U` (postmaster socket only) → reported healthy mid-`initdb` → first `psql` connect failed under `sh -ec` → exit 2 (NB: psql exit 2 = connection, exit 3 = SQL error). Fix: healthcheck `-d $DB_DATABASE` + `start_period:40s`; init entrypoint has a **wait-for-postgres retry loop** (60×2s) before seeding.
3. **`dash_data_sources` UndefinedTable WARNING every boot** — vestigial parent-Dash table never created here; `init_connectors` ALTERed it blindly. Fix: `to_regclass()` guard → skip when absent (connectors feature is off in this fork anyway).

### Apache AGE graph (2026-06) — real openCypher on cp-db
Pharma drug-relationship graph for substitution / alternative questions that plain SQL handles poorly.
- **Install (non-destructive):** AGE built from `apache/age` master against PG18.4 (arm64) inside cp-db (apt build-essential/flex/bison/postgresql-server-dev-18 → `make install` → `age.so` in `/usr/lib/postgresql/18/lib`). `CREATE EXTENSION age;` + `ALTER SYSTEM SET shared_preload_libraries='age';` + **one cp-db restart** (data in PGDATA volume `/var/lib/postgresql/18/docker`, safe). pgvector coexists.
- **pgbouncer-safe runtime pattern:** app connects via PgBouncer transaction-mode where session `LOAD`/`SET` don't persist. With `shared_preload='age'`, no LOAD needed; queries still need `ag_catalog` in search_path → wrap each graph call in a **transaction with `SET LOCAL search_path = ag_catalog, "$user", public`**. The tool sidesteps this entirely by opening its **own direct read-only connection to cp-db (service `dash-db:5432`, NOT pgbouncer)** per call.
- **Graph `citypharma`** (built by `scripts/build_pharma_graph.py`, run in cp-api → direct cp-db): nodes Article(4892)/Generic(1001)/Category(101)/Indication(1523)/Composition(2323); edges HAS_GENERIC/IN_CATEGORY/TREATS/HAS_COMPOSITION + **SUBSTITUTE_OF (41,042**, derived: same generic_name). **Stock stays relational** (`citypharma_balance_stock` 106k) — joined by `article_code` at query time (hybrid: graph for relationships, SQL for the 106k stock facts).
- **Agent tools** (`dash/tools/pharma_graph_tool.py`, wired in `dash/tools/build.py`, gated `PHARMA_GRAPH_DISABLED`): `find_substitutes(article_code|brand_name, site_code, in_stock_only)`, `alternatives_for_indication(indication, …)`, `drug_relationships(…)`. Instruction nudge in `instructions.py` ("💊 PHARMA KNOWLEDGE GRAPH"). **Verified live:** chat "substitutes for AMOR LEVONORGESTREL 0.75G" → agent called `find_substitutes` → "Three Levonorgestrel alternatives in stock: ECEE 2 (0.75mg), LYDIA 1 (1.5mg)…".
- **DURABILITY LANDMINE:** `age.so` is in the cp-db container fs (ephemeral), and `shared_preload='age'` is in PGDATA. `docker restart cp-db` keeps the fs (safe). ### Pharma `article_code` — cross-table type mismatch + source corruption

### Embed store-scope isolation (2026-06-08) — 3 fixes, security-critical
A store-bound embed (`bound_scope_id`) must answer ONLY its own store. Three bugs let it leak other stores' data (cross-store, 12%→95% correct after fix):
- **Team cache MUST be per-store.** `store_id` is baked into the prompt at team-build time; the embed now passes the per-embed `synthetic_viewer` (`-embed_pk`) as `user_id` to `create_project_team` so each store gets its own cached team. `user_id=None` collides on `citypharma_None_<lang>` → first store's prompt reused for all (60s TTL). - **Embeds MUST set the SQL scope.** The embed `_embed_user` needs `"via_api_key": True` or `resolve_api_scope()` returns None → `API_STORE_SCOPE` None → stock tool drops `WHERE site_code=…` → global-total leak. With it set, the SQL filter enforces the store regardless of the prompt (defense-in-depth).
- **`store_stock_summary` uses LEFT JOIN, not INNER.** `stock_qty` is in the stock table alone; the article join only enriches `unique_articles`. INNER drops every row when `article_code` is the 1E+12 corruption (text≠bigint::text) → total 0. (low_stock stays INNER — it `int()`s article_code.)
- Embed concurrency is NOT serialized (no single-agent lock like the gateway) — gated by `llm_parallel_cap_chat` (live UI setting, v1.18.0; seeded 40, semaphore rebuilds on change). ### Training pipeline + data rules (2026-06-09, latest+49)
- **Aggregate in SQL, never in-head.** `dash/instructions.py` has a HARD RULE: totals/counts/distinct MUST come from SQL `SUM`/`COUNT(DISTINCT)`/`GROUP BY`, never by summing fetched rows (the agent used to undercount many-row queries). Prefer the pre-aggregated `shop_flat` for branch-stock totals.
- **Code/ID columns land as TEXT.** `_is_id_colname` (`app/upload.py`) normalizes separators (`"Article Code"` → `article_code`) BEFORE the id/code regex, so codes never land as int64 and cross-table joins don't hit the bigint↔text mismatch. Never raw-compare `article_code` across tables — always `::text`, or use `shop_flat` (already normalized).
- **Derived tables are NOT trained.** `shop_flat` is excluded from the training table list (`_DERIVED_TABLES` in `app/upload.py`); it's built by `build_shop_flat` in post-hooks, not trained. `STOCK_COLS=("site_code","stock_qty","article_code")` in `dash/tools/table_sync.py` — `article_code` is REQUIRED so the resolver can never pick `shop_flat` (which keys on `art_key`) as the stock table.
- **Training step timeouts + watchdog.** Relationship/QA verify engines run with `statement_timeout=30s` (`connect_args options`) so an un-indexable `::text` JOIN can't hang a run. `_reap_stale_runs(slug)` (`app/learning.py`) runs on every status poll: any `running`/`finalizing` run with no log progress > `STALE_RUN_MINUTES` (default 12) is auto-failed → UI spinner can't spin forever. Run status: `running` → `finalizing` (post-hooks) → `done`.
- **No ML step.** AutoML removed 2026-05-23; `_task_ml()` is a no-op that PURGES any stale `ml_auto_create` row, frontend filters `ml`/`ml_auto_create`/`ml_models`. No "ML Models" badge, no machine-learning training in this product.
- **`APIGW_AUTO_PROVISION=0`** (default off, `.env` + compose). Loading multi-store stock no longer auto-mints `svc:outlet-*` API service accounts. Turn on only for per-store gateway keys.
- **Fresh-DB baseline** (`db/baseline/schema.sql` recon block): runs under `SET search_path = public, dash;` + `ADD COLUMN IF NOT EXISTS engine_model` on usage tables before the `v_usage_unified` recreate — required or a fresh deploy fails to load. Locked `dash_projects` row still seeded by hand on a fresh DB (`ensure_locked_project()` not built).

### Embed widgets INHERIT the Brand theme — snippet must not freeze appearance (2026-06-11, `1.13.0`)
